# Pure ORM 设计思想
## 一、框架核心定位
Pure ORM 秉持**极简、纯粹、透明、可控**的核心理念，彻底剥离传统持久层框架所有冗余设计，回归 ORM 最原始、最本质的核心职责。
仅专注两大核心能力：**基于Lambda类型安全编码构建 SQL**、**基于Lambda类型安全实现全自动数据库结果与实体对象映射**。
不做过度封装、不搞复杂抽象、不附加非核心功能，不追求全方言兼容、不追求冗余规范全覆盖，专注解决生产环境真实痛点，轻量化、低内存、易调试、易维护，真正做到简洁、实用、高效。

同时框架**兼容原生 SQL 编写**，保留最灵活的自定义 SQL 能力，所有原生 SQL 均可享受全自动映射，不强制、不限制、不抛弃原生语法，满足极端复杂场景需求。日常开发无需手写冗余原生 SQL，兼顾灵活与高效。

## 二、全面摒弃传统框架糟粕
### 1、摒弃 过度抽象、深度封装、黑盒运行
彻底抛弃传统ORM框架**层级极深、过度抽象、高度封装**的劣质设计。
传统ORM框架底层逻辑复杂、嵌套繁琐、执行流程全封闭，开发者无法查看、无法追踪、无法断点调试，线上问题定位难、维护成本高。
Pure ORM 全程代码可见、逻辑透明、无隐藏逻辑、无底层代理、无复杂嵌套，每一行 SQL、每一步映射都可追踪、可调试，彻底告别黑盒开发。

### 2、摒弃 多级缓存、内置缓存耦合设计
直接删除框架内置一级缓存、二级缓存、全局缓存等所有冗余设计。
传统框架自带缓存，极易造成数据不一致、缓存失效、脏数据、查询异常，且缓存与 ORM 核心逻辑强耦合，问题难排查、逻辑难控制。
Pure ORM 坚持职责分离，**缓存交由专业缓存框架实现**，ORM 只专注数据映射，不耦合、不冗余、不产生隐式问题。

### 3、摒弃 懒加载、级联加载、级联操作
彻底移除懒加载、一对多级联、多对多级联、隐式查询等所有不稳定设计。
传统框架自动触发级联查询、懒加载 SQL，极易引发性能灾难、内存溢出、查询不可控，前后端分离架构下异常频发。
Pure ORM 无任何隐式查询、无自动关联、无懒加载，所有查询逻辑由开发者显式控制，稳定、安全、无意外。

### 4、摒弃 XML 配置、字符串 SQL、HQL 语法
抛弃 MyBatis 大量 XML 配置、SQL 与代码分离；抛弃 Hibernate HQL 字符串、JPA 复杂注解。
告别配置散落、SQL 硬编码、类型不安全、编译不报错、运行才出错的问题。
全程基于 **Lambda 表达式** 编码开发，类型安全、编译检查、无字符串、无配置文件，开发更稳健、更高效。

### 5、摒弃 臃肿冗余功能、非本职扩展能力
抛弃复杂事务封装、复杂方言兼容、复杂生命周期、复杂代理对象等所有无用功能。
传统框架大而重，功能泛滥、内存占用高、启动缓慢、容器部署臃肿。
Pure ORM 只保留核心能力，极简轻量化，无多余对象、无多余逻辑，运行高效、内存友好，适配云原生、微服务、高并发场景。

### 6、摒弃 复杂多对多映射、冗余实体设计
传统框架支持复杂多对多映射，配置繁琐、逻辑混乱、查询不可控，极易产生冗余 SQL 与逻辑漏洞。
Pure ORM 只保留**一对一、一对多**标准关联映射，多对多通过SQL语句join中间表简化处理，实体层干净简洁，无复杂配置、无映射陷阱。

## 三、Pure ORM 自身干净设计原则
1. 全程 Lambda 编码，类型安全，无字符串 SQL，无冗余配置
2. 支持原生 SQL，同时全自动映射结果，灵活与高效兼顾
3. 原生支持生产必备语法：select for update 行锁、upsert 新增或更新
4. 无黑盒、无黑魔法、无隐式逻辑，全程可调试、可维护
5. 只做 ORM 核心工作，不冗余、不臃肿、不耦合、不侵入
6. 简单易懂、上手低成本、线上稳定、排错简单
7. 仅支持标准实体属性与数据库适配类型，涵盖数据基本包装类、LocalDate、LocalDateTime 等标准日期时间类型，同时支持枚举类型，不兼容复杂自定义类型与非标准属性扩展，实体结构极简稳定
## 四、整体设计总结
Pure ORM 不是堆砌功能，而是**做减法、去糟粕、留本质**。
把传统 ORM 所有复杂、臃肿、不可控、难调试、高耦合的设计全部剔除，只保留最干净、最核心、最实用的能力。
以极简架构、透明逻辑、编码式开发，真正做到轻量、稳定、高效、可控，是适配现代开发架构的**纯粹原生 ORM 框架**。
